# Policies & Procedures Reference

## Required Policy Set

Every SOC2 engagement requires these documented policies. They don't need to be long — a clear, enforced 2-page policy beats an ignored 30-page one.

| Policy | TSC Mapping | Key Contents |
|--------|-------------|-------------|
| Information Security Policy | CC1, CC5 | Scope, roles, responsibilities, acceptable use |
| Access Control Policy | CC6 | Authentication, authorization, review, revocation |
| Change Management Policy | CC8 | Request, approval, testing, deployment, rollback |
| Incident Response Policy | CC7 | Classification, response, communication, post-mortem |
| Data Classification Policy | C1 | Classification levels, handling requirements |
| Backup & Recovery Policy | A1, CC9 | Backup schedule, retention, restore testing |
| Vendor Management Policy | CC9.2 | Assessment, contractual requirements, monitoring |
| Acceptable Use Policy | CC1, CC2 | Employee responsibilities, prohibited actions |
| Privacy Policy | P1–P8 | Collection, use, retention, rights (aligns with GDPR) |
| Business Continuity Policy | CC9 | BCP, DR plan, RTO/RPO definitions |
| Risk Management Policy | CC3 | Risk assessment methodology, treatment options |
| Encryption Policy | CC6.7, C1 | Encryption standards, key management |

## Policy Template Structure

```markdown
# [Policy Name]

**Version:** [X.Y]
**Owner:** [Role, not person name]
**Approved by:** [Role]
**Effective date:** [YYYY-MM-DD]
**Next review:** [YYYY-MM-DD] (annual minimum)

## Purpose
[Why this policy exists — 2-3 sentences]

## Scope
[What systems/people/data this covers]

## Policy Statements
[Numbered, clear, actionable requirements]

## Roles and Responsibilities
[Who does what]

## Exceptions
[How to request exceptions, who approves, documentation required]

## Enforcement
[Consequences of non-compliance]

## Revision History
| Version | Date | Author | Changes |
|---------|------|--------|---------|
```

**Tips:**
- Use version control (Git) for policies — this gives you change history for free
- Policies must be **acknowledged** by employees — track signed acknowledgments
- Review annually at minimum, and after any significant incident or organizational change

## Change Management

### Change Types

| Type | Approval | Lead Time | Examples |
|------|----------|-----------|---------|
| Standard | Pre-approved | None | Dependency update, config tweak |
| Normal | Peer review + lead | 1+ business day | Feature, refactor, infra change |
| Emergency | Post-hoc review | Immediate | Security patch, production outage fix |

### Change Request Process

```
1. SUBMIT    → Developer opens PR / change ticket
2. REVIEW    → Peer code review (min 1 reviewer, 2 for High/Critical)
3. ASSESS    → Risk/impact assessment (automated + manual for High/Critical)
4. APPROVE   → Engineering lead approves (+ security review if needed)
5. TEST      → Verified in staging environment
6. DEPLOY    → Deployed via CI/CD pipeline (no manual deploys)
7. VERIFY    → Post-deployment checks (health, metrics, alerts)
8. RECORD    → Change logged with all artifacts
```

### CI/CD Compliance Requirements

These controls must be enforced in the pipeline — not as guidelines but as gates:

```yaml
branch_protection:
  main:
    require_pull_request: true
    required_reviewers: 1          # 2 for critical systems
    require_status_checks:
      - tests-pass
      - security-scan-pass
      - lint-pass
    dismiss_stale_reviews: true
    restrict_push: true            # No direct pushes
    require_linear_history: true   # No merge commits (audit clarity)

deployment:
  staging:
    trigger: merge-to-main
    automatic: true
  production:
    trigger: manual-approval       # Human gate
    requires: staging-verification-pass
    rollback: automated            # One-click rollback capability
```

### Evidence Generated by Change Management

Auditors will sample changes and expect to see:
- PR with code review comments and approval
- Associated ticket/issue (traceability)
- CI/CD pipeline run showing all gates passed
- Staging test results
- Production deployment log
- Post-deployment verification

**Anti-pattern:** Approving your own PR. Even solo developers should use branch protection and delay merging (or have an external reviewer — a contractor, advisor, or automated review tool).

## Incident Response

### Severity Matrix

| Severity | Criteria | Response Time | Notification | Post-Mortem Required |
|----------|----------|---------------|-------------|---------------------|
| SEV-1 Critical | Data breach, full outage, security compromise | 15 min to acknowledge | Customers, leadership, possibly regulators | Yes, within 48 hours |
| SEV-2 Major | Significant degradation, partial breach risk | 1 hour | Internal stakeholders, affected customers | Yes, within 1 week |
| SEV-3 Moderate | Minor feature impact, no data risk | 4 hours | Internal team | Optional |
| SEV-4 Low | Cosmetic, internal tooling | Next business day | Ticket only | No |

### Incident Commander (IC) Responsibilities

- Owns the incident from triage to closure
- Makes containment decisions (rollback, isolate, communicate)
- Coordinates responders, avoids conflicting actions
- Provides status updates on defined cadence
- Ensures post-mortem is completed and action items tracked

## Vendor Management

### Vendor Risk Tiers

| Tier | Criteria | Assessment | Review Frequency |
|------|----------|-----------|-----------------|
| Critical | Handles customer data, infrastructure dependency | Full security assessment, SOC2/ISO27001 required | Annual |
| High | Access to internal systems, no customer data | Security questionnaire, SOC2 preferred | Annual |
| Medium | SaaS tool, limited data exposure | Security page review, DPA | Biennial |
| Low | No data access, replaceable | Basic due diligence | On renewal |

### Vendor Assessment Checklist

- [ ] SOC2 Type II report reviewed (or ISO 27001 cert)
- [ ] DPA / data processing agreement executed
- [ ] Sub-processor list reviewed
- [ ] Data residency confirmed (EU if required)
- [ ] Incident notification clause included (≤72 hours)
- [ ] Right to audit clause included
- [ ] Exit/data portability provisions documented
- [ ] Insurance coverage verified (for Critical tier)
- [ ] Security questionnaire completed (SIG Lite or custom)

### Vendor Register

Maintain a register of all vendors with:
- Vendor name, service, tier, data types shared
- Contract dates (start, renewal, termination notice period)
- Security assessment date and result
- DPA status
- Primary contact
- Sub-processors

## Security Awareness Training

### Training Requirements

| Audience | Topics | Frequency | Evidence |
|----------|--------|-----------|----------|
| All employees | Phishing, passwords, social engineering, data handling, incident reporting | Annual + onboarding | Completion certificate, quiz score |
| Developers | Secure coding (OWASP Top 10), secret management, dependency security | Annual | Training record, code review metrics |
| Admins | Privileged access, key management, infrastructure hardening | Annual | Training record |
| Leadership | Risk oversight, compliance obligations, incident escalation | Annual | Acknowledgment |

### Training Evidence

- Track completion in an LMS or spreadsheet with: name, date, course, score
- Keep records for the full observation period + 1 year
- New hires must complete training within 30 days of start
- Remedial training for employees who fail phishing simulations
