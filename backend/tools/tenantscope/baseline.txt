# tenantscope analyzer baseline — pre-existing violations exempted from CI.

# Each line is a tenant-isolation violation the analyzer found but that
# predates the analyzer itself. As each module migrates to use
# shared/tenantrepo.Scope, delete the corresponding lines from this file.
# CI fails if the analyzer finds any violation NOT listed below, so new
# violations are caught even while the migration is incomplete.
#
# Phase 5.4 expanded the scan scope from internal/addons/ to also cover
# internal/core/ and internal/shared/. The new entries below capture the
# existing state of those packages — many of which manage platform-global
# data (users, sessions, modules) and will stay baselined indefinitely.
# The right long-term home for those is a //tenantscope:allow admin-view:
# comment at the call site; converting them is ongoing work.
#
# Format: <path relative to backend/>:<line>:<method>
#
# To regenerate the baseline (e.g. after legitimate movement in existing
# files), run from backend/:
#   go run ./tools/tenantscope/cmd/tenantscope ./internal/... \
#     | awk -F': tenantscope: ' 'NF==2 { split($1, a, ":"); p=a[1]; l=a[2]; \
#         sub(/^\/app\//, "", p); n=split($2, w, " "); m=w[1]; print p":"l":"m }' \
#     | sort -u > tools/tenantscope/baseline.txt
#
# Do not add new entries to this file by hand. If you have a genuinely
# legitimate need to bypass (e.g. a reaper job that operates across all
# tenants), use a //tenantscope:allow <reason> comment at the call site.
# Expiry is optional: //tenantscope:allow-until=YYYY-MM-DD <reason>
# makes the allowance auto-expire once the date passes, so temporary
# workarounds cannot pile up in perpetuity (Phase 5.4).
