# Security Policy

## Reporting a vulnerability

If you believe you have found a security issue in Orkestra, **please report it privately** so we can investigate and ship a fix before details become public. Do not open a public GitHub issue.

**Contact:** <salvatore.balestrino@gmail.com>

Please include:

- A description of the issue and its impact
- Steps to reproduce (or a proof-of-concept)
- The affected component (`backend`, `frontend-admin`, `frontend-client`, `mobile`, `docker`, or a specific module)
- The version / commit SHA you tested against
- Any suggested mitigation, if you have one

We aim to:

| Stage | Target |
|---|---|
| Acknowledge the report | within **72 hours** |
| Confirm the issue (or rule it out) | within **7 days** |
| Ship a fix for confirmed high-severity issues | within **30 days** |
| Public disclosure | coordinated with the reporter, after a fix is available |

Researchers acting in good faith under this policy will not be pursued legally.

## Scope

Orkestra is a modular monolith plus optional addon modules. The following are in scope:

- The Go backend (`backend/`) — including all core and addon modules, the AI service sidecar, and shared infrastructure (auth, RBAC, tenant isolation, secrets handling).
- The React admin SPA (`frontend-admin/`) and client SPA (`frontend-client/`).
- The Flutter mobile app (`mobile/`).
- CI/CD configuration (`.github/workflows/`) — supply-chain issues, exposed secrets, unsafe action versions.
- Docker images published to GHCR (`ghcr.io/orkestra-cc/orkestra/*`).
- Example deployment configurations under `docker/`.

The following are **out of scope** (please do not submit reports for these):

- Issues in upstream dependencies that are already published and have no Orkestra-specific exploit path (report those upstream instead).
- Best-practice findings without a concrete attack scenario (e.g., "you should add header X" without an exploit).
- Self-XSS that requires the victim to paste attacker-controlled JavaScript into their own console.
- Volumetric DoS / rate-limiting findings against demo or self-hosted instances.
- Reports auto-generated by scanners with no manual validation.

## Severity guidance

When triaging, we use the following rough scale:

- **Critical** — pre-auth RCE, full multi-tenant data crossover, complete authentication bypass, secret exfiltration from `ConfigService`.
- **High** — authenticated RCE, single-tenant data crossover, privilege escalation across the two-tier tenancy boundary (Tier-1 ↔ Tier-2), session takeover.
- **Medium** — stored XSS, IDOR within a single tenant, CSRF on authenticated state-changing endpoints, audit-log evasion.
- **Low** — reflected XSS in admin-only views, information disclosure of non-sensitive metadata, missing security headers without a specific exploit.

## Disclosure

Once a fix is shipped, we credit the reporter (with their consent) in the release notes and in [`SECURITY-ADVISORIES.md`](docs/SECURITY-ADVISORIES.md) — or in the public GitHub Security Advisory, if the issue warranted a CVE.

## Supported versions

Until a stable `v1.0.0` release, only the `main` branch is supported for security fixes. After `v1.0.0`, this section will be updated with a version-support table.

| Version | Security fixes |
|---|---|
| `main` (rolling) | ✅ |
| pre-`v1.0.0` tags | ❌ — upgrade to `main` |
